1. WHAT WE ARE BUILDING (current scope)
------------------------------------------------------------------------
A mobile-friendly WEBSITE (not a native app) for collecting photos from
guests' phones.

User flow:
  1. A guest scans a QR code.
  2. The QR opens the website in their phone browser (works on iOS and
     Android).
  3. The page shows a big primary button in POLISH that opens the
     phone's native photo library / picker.
  4. The guest selects one or more photos.
  5. Selected photos are shown as thumbnail previews on the page.
  6. A SEND button is ALWAYS visible, fixed to the bottom of the screen.
  7. Tapping SEND uploads the selected photos (to a backend we add
     later) with a progress indicator and a success confirmation.

This pattern is commonly used for weddings / parties / events where many
guests contribute photos to one shared collection.

3. IMPORTANT TECHNICAL CONSTRAINT (read this)
------------------------------------------------------------------------
GitHub Pages can HOST the website, but it CANNOT STORE uploaded files.
  - GitHub Pages is static hosting: it only serves HTML/CSS/JS.
  - It has no server-side code, so it cannot receive or save uploads.
  - A git repo is not writable by website visitors, and committing
    guests' photos into a (public) repo would be wrong anyway.

Therefore, in the current phase:
  - The website is fully built and testable.
  - The SEND button is wired to a CLEARLY-MARKED PLACEHOLDER upload
    function (a single, isolated function in the code).
  - When the real server exists, we change only that one function.

What "works" now vs. later:
  - WORKS NOW: QR -> page loads -> Polish upload button -> native photo
    picker -> multi-select -> thumbnail previews -> sticky SEND button
    -> progress UI -> success/error messages.
  - NOT YET: photos actually being stored somewhere. That needs the
    backend (section 4).

4. PLANNED NEXT PHASE - ON-PREM SERVER (to do later)
------------------------------------------------------------------------
You will set up an on-prem (home) server later and ask Claude to "plug
it in." When that happens, we will need to decide/answer:

  - Server type/OS (NAS, Raspberry Pi, old PC, Linux/Windows, Docker?).
  - Public reachability: phones uploading over cellular need the server
    reachable from the internet (port forward + dynamic DNS, a tunnel
    like Cloudflare Tunnel/Tailscale Funnel, or a reverse proxy).
  - HTTPS certificate (browsers + camera/file APIs strongly prefer
    HTTPS; mixed content from an https page to http server will fail).
  - The upload endpoint contract (e.g. POST multipart/form-data to
    /upload, returns 200 on success).
  - Where files land on disk and how they're named/organized
    (e.g. one folder, or by date, to avoid filename collisions).
  - CORS: the server must allow uploads from the GitHub Pages origin.
  - Basic abuse protection (see section 5).

Integration point in the code: the single placeholder upload function.
Only its body (the actual network call to the server) needs to change.

6. TECH PLAN (proposed - not yet built)
------------------------------------------------------------------------
- Plain static site: index.html + styles + a small script. No framework
  needed for v1 (keeps it simple and GitHub-Pages-friendly). Could use a
  framework later if desired.
- Native picker via:  <input type="file" accept="image/*" multiple>
  (Optionally capture from camera too; multi-select from library is the
  priority.)
- Thumbnail previews via object URLs in the browser.
- Sticky SEND button via fixed/sticky CSS positioning at viewport bottom.
- Upload via fetch() multipart/form-data inside the ONE placeholder
  function (currently stubbed/simulated).
- Deploy: push to a GitHub repo, enable GitHub Pages -> get the
  https://<user>.github.io/<repo>/ URL -> make the QR from that URL.
